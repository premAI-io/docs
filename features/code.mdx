Connect GitHub and Fluso starts behaving like a developer who already knows your codebase. The point isn't that it can write code in the abstract — most things can. The point is that it reads your repos first and writes things that fit them. The PR it opens uses your patterns. The bug it finds points at the actual file and line.

What follows is what it does, with concrete examples. The pattern is consistent: describe an outcome, Fluso does the work, you review the diff before anything is filed.

## Reading a codebase

The most common entry point is asking how something works:

> *"Show me how authentication works in this repo."*

> *"Find where JWT verification happens."*

> *"What's the architecture of the backend? How are services organised?"*

> *"Where are the database models defined?"*

> *"What middleware does each request go through?"*

The replies aren't generic explanations of authentication. They're explanations of authentication in your code, with file paths and line numbers. Useful for the kind of codebase you used to know and now don't quite, or for joining a project mid-stream.

## Reviewing pull requests

PR review is where the time savings are most visible:

> *"Review the open PR from Sarah. Security, edge cases, test coverage."*

What comes back is a real review, not a "looks good to me". Summary of what the PR does. Issues flagged with severity: yellow for "address before merge", red for "this is a security hole", green for "no concerns". Suggestions for things that aren't blockers but would improve the patch. A verdict.

For a rate-limiting PR it might look like: *"Adds rate limiting middleware for the API. Three files changed, 85 lines added. Issues: missing rate limit on /auth/login (brute-force vulnerability); reset window of 1 hour might be too long, 15 minutes seems more standard. Suggestion: add a test for the rate-limit-exceeded response shape; consider making the limit env-configurable. Verdict: approve with suggestions, but address the missing login limit before merge."*

Other useful prompts:

> *"Summarise what changed in PR #42."*

> *"Are there PRs that need my review?"*

> *"Which open PRs are safe to merge?"*

The intent is to let humans focus on architecture and design. Fluso catches mechanical stuff: null checks, error handling gaps, missing tests, inconsistent patterns, security regressions.

## Bug fixes

The pattern: describe the symptom, get the diagnosis, optionally let Fluso write the patch.

> *"Users report password reset emails aren't arriving. Find the bug in our auth module."*

Fluso traces through the code from request entry to email send, identifies the issue, and tells you the file and line. *"Found it. The password reset email template in `templates/reset-password.html` line 12 uses `https://old-domain.com/reset?token=...`. The base URL changed when we migrated to the new domain last week. The links in the emails are broken."*

If you want it fixed:

> *"Fix it and create a PR."*

You get back: *"Done. PR #156 — fixed the reset link domain, added a test that validates the link format. Ready for review."*

You read the diff before merging. Fluso always asks before opening a PR.

## Scaffolding projects

If you need a new service or app from scratch:

> *"Build me a REST API for a todo app using FastAPI with SQLite. User auth, CRUD endpoints, tests, API docs."*

> *"Create a Next.js landing page with a blog, contact form, dark mode, Tailwind."*

> *"Scaffold a Go microservice with gRPC endpoints, Postgres, Docker."*

What you get is a complete project: structure, working code, database models and migrations, endpoints with validation, tests (unit and integration), a README that walks setup, Docker config, CI pipeline. It runs.

Stacks Fluso scaffolds well: Next.js + TypeScript, FastAPI + Python, Go + Connect RPC, Rails + Ruby, Rust, Flutter, React.

## Writing code

The thing that matters here is that Fluso reads your existing code first and follows the patterns it finds:

> *"Add pagination to /users. Same pattern as /products."*

> *"Write a database migration to add a 'role' column to the users table."*

> *"Add input validation to create user. Email format, password strength, required fields."*

> *"Write unit tests for the payment processing service."*

The output won't have a different style from the rest of the codebase. If your repo uses single quotes and no semicolons, the new code does too. If your tests use a specific helper, Fluso uses it. If you have a particular service-layer convention, the new code lives in the right place.

Drop a `context.md` in your workspace if your conventions are subtle:

```markdown
# Project context

- Stack: Next.js 14, TypeScript, Postgres, Drizzle ORM
- Style: 2-space indent, no semicolons, single quotes
- Testing: Vitest, Playwright for e2e
- CI: GitHub Actions, must pass type-check + lint + tests
- Conventions: route handlers in app/api, services in lib/services
```

Fluso reads it and follows.

## CodeTours

Useful for onboarding:

> *"Create a CodeTour of the authentication module for new team members."*

You get a `.tour` file: a step-by-step walkthrough that links to specific files and line ranges with explanations.

1. `middleware/auth.ts:12-45` — How requests are intercepted and JWT tokens extracted.
2. `services/jwt.ts:23-67` — Token verification and user lookup.
3. `handlers/login.ts:8-34` — Login flow and credential validation.
4. `models/user.ts:5-29` — User model and database schema.

A new hire can step through it instead of reading the code blind for two days.

## Issues

> *"List the open issues on fluso-frontend."*

> *"Create an issue for the login bug. Use the bug report template."*

> *"What issues are assigned to me this sprint?"*

> *"Close issue #142. Fix is deployed."*

All standard, all conversational.

## Where the savings show up

PR review: Fluso catches mechanical issues so humans focus on architecture and design.

Bug-to-fix loops: a 2-4 hour debug becomes a 15-minute exchange.

Onboarding: new hires ramp in hours, not weeks, because they can ask the codebase questions in plain English instead of reading 30,000 lines blind.

<Warning>
  Fluso always asks before modifying repositories. PRs, commits, and issue closes wait for explicit approval. The diff or comment is shown for review every time.
</Warning>
