Each app you connect gives Fluso more to work with. More context, so it understands what you mean. More capability, so it can act on what you ask. The catalog is over a hundred services; the four below cover most people's day.

This page walks through them one at a time. For the full list, see [All apps](/integrations/all-apps).

## How it works under the hood

You click **Connect** on a card in **Settings → Connectors**. The app's own OAuth screen opens. You see the exact permissions Fluso wants and approve the ones you're comfortable with. After that, Fluso can use the app inside chat. There's no key to copy, no callback URL to set, no shared password anywhere.

Tokens live encrypted in a dedicated secrets store. You can revoke access from Fluso or from the source app's own settings. Disconnecting deletes the token immediately.

## Where to start

Connect things in the order your work demands them, not all at once.

Gmail first. It's the highest-leverage connection because email is where most work originates, which means it's where Fluso starts to feel useful fastest.

Calendar second. Together with Gmail, you have the morning brief and meeting prep working.

Slack only if your team actually lives there. If most coordination happens in email, skipping Slack costs nothing.

GitHub only if you write code. The features it unlocks are powerful, but they're irrelevant if you don't.

## Gmail

Click **Connect** on the Gmail card. Sign in to Google with the account you want Fluso to assist with.

The three permissions:

| Permission | Why it's there |
|---|---|
| Read messages | Summaries, search, task extraction. |
| Send messages | Reply drafting. Sending always waits for your approval. |
| Read headers | Categorisation by sender, label, date. |

Notably absent: delete or modify account settings. Fluso has no path to either.

Setup is a click and a confirmation screen.

## Google Calendar

Same flow as Gmail. If you authorised Gmail with the same Google account, the calendar prompt is even shorter.

Two permissions: read events (so Fluso can see your schedule) and create or modify events (so it can schedule meetings when you ask). Without write access the morning brief still works, but scheduling a meeting from chat doesn't.

## Slack

The Slack card opens an authorisation screen scoped to a workspace. Pick the workspace; you can connect more than one if you split work across them.

You also choose channel scope: public channels, a hand-picked list of private ones, or every channel you have access to. Most people start narrow and widen later.

Permissions: read messages in authorised channels, post messages with your approval. Posting drafts behaves like email — Fluso shows the message, you approve, then it sends.

## GitHub

GitHub's OAuth flow lets you grant access to specific repositories or to all of them. Either is fine. You can change the set later from your GitHub settings without reconnecting in Fluso.

Permissions: read on repos, read/write on issues and PRs. Every PR or comment Fluso creates appears for review before it's filed.

## Managing what's connected

Open **Settings → Connectors** and you can see what's connected, when each token was last refreshed, and what each app has access to. Disconnect from there too. Audit it once a month — fewer well-used connections beat many half-forgotten ones.

## Privacy questions that come up

<AccordionGroup>
  <Accordion title="What does Fluso actually store?">
    OAuth tokens, encrypted, in a dedicated secrets manager. Conversations, files, and the knowledge graph live in your isolated workspace. Passwords are never seen.
  </Accordion>
  <Accordion title="Can it act without me approving?">
    No. Reads happen freely. Anything that writes to another system — sending a message, opening a PR, creating an event — waits for your explicit approval first.
  </Accordion>
  <Accordion title="What happens when I disconnect?">
    The token is deleted. Fluso loses access immediately. Anything already in your knowledge graph stays unless you delete it.
  </Accordion>
  <Accordion title="Can teammates see my apps on a Team plan?">
    Admins can see which apps each member has connected so they can manage access. They cannot read the data.
  </Accordion>
</AccordionGroup>

For the longer version, see [Security & privacy](/resources/security).
